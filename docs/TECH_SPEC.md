# Техническая спецификация и архитектура

## Назначение и границы системы

Бот принимает заявки на бесплатный осмотр автомобиля, передаёт их сотрудникам студии и сопровождает запись до завершения визита. PostgreSQL — единственный источник истины для бизнес-данных; Redis используется для FSM, ограничения частоты и временных удержаний; Telegram не является хранилищем состояния.

Время в таблицах хранится в UTC. Перед показом пользователю и сотруднику оно преобразуется в `STUDIO_TIMEZONE`. Все идентификаторы доменных сущностей — UUID; Telegram ID хранится отдельным уникальным bigint-полем.

| Администратор | Просмотр очереди, подтверждение/отклонение, предложение времени, работа с расписанием, услугами и FAQ | Проверяется каждый update: Telegram ID обязан входить в `ADMIN_IDS`. Не может действовать от имени клиента и не видит обращения, не связанные с рабочими задачами. | `/admin` → список → карточка → решение/контент/расписание; решение возвращает к карточке или списку. |

## Модульная архитектура

| Слой | Ответственность |
| --- | --- |
| `app/bot/handlers` | Получает Telegram updates, валидирует интерфейсный ввод, вызывает сервисы; бизнес-правил не содержит. |
| `app/bot/keyboards`, `states`, `filters`, `middlewares` | Типизированные callback, FSM, авторизация, сессия БД, логирование, rate limit. |
| `app/services` | Транзакционные сценарии: запись, перенос, отмена, расчёт, уведомления, контент. |
| `app/repositories` | Изолированные SQLAlchemy-запросы и блокировки строк. |

| `app/scheduler` | Восстановление и доставка уведомлений через APScheduler. |
| `app/config`, `utils` | Pydantic Settings, часовые пояса, телефон, корреляция и структурированные логи. |

## Жизненный цикл и отказоустойчивость

При старте конфигурация валидируется, открываются async-подключения PostgreSQL/Redis, регистрируются роутеры и запускается scheduler. На остановке scheduler прекращает принимать задания, polling останавливается, затем корректно закрываются Redis, Bot session и SQLAlchemy engine.

Повторные callback обрабатываются идемпотентно: клиентская отправка имеет `idempotency_key`, решение администратора допускает повтор только в тот же статус, уведомление имеет уникальный `idempotency_key`. При временном сбое Telegram уведомление переводится в retry с ограниченным количеством попыток; при блокировке бота — в failed без повторов. После перезапуска scheduler загружает из БД все due notifications, поэтому задания не теряются.

## Безопасность и данные

Телефон принимается кнопкой контакта или текстом, нормализуется в E.164 и валидируется библиотекой `phonenumbers`. Фото: максимум `MAX_PHOTOS_PER_BOOKING`; разрешены Telegram Photo и документы JPEG/PNG/WEBP, размер не выше `MAX_PHOTO_SIZE_MB`; видео, архивы, исполняемые файлы и дубликат `unique_file_id` отклоняются. Локальное скачивание не требуется.

Все запросы выполняются ORM/параметрами. Администраторские handlers защищены фильтром на каждом update. Логи имеют correlation ID, не содержат токен и полный телефон. Ошибки пользователю формулируются без SQL, путей, stack trace и внутренних ID. Секреты существуют только в `.env`, который исключён из репозитория.

## Конкурентность записи

Слот сначала получает временный hold с UTC-expiration. Перед финальной отправкой сервис в одной транзакции блокирует строку `TimeSlot`, проверяет owner/срок/статус, создаёт Booking и переводит слот в booked. Уникальная связь active booking ↔ slot и обработка `IntegrityError` являются второй линией защиты от гонки. Истёкшие hold очищаются фоновым процессом; hold никогда не считается подтверждённой записью.
